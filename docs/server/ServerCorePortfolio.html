2 프로젝트 범위

2.1 왜 Runtime core를 분리했나

초기 서버 구현은 네트워크 런타임과 게임 서버 도메인이 한 계층에 섞여 있었습니다. IoRing, Session, BufferPool 같은 범용 I/O 구조 옆에 Room, PlayerId, PacketSession 같은 게임 전용 구조가 함께 놓이면서, HTTP 서버나 TCP proxy처럼 게임과 무관한 서버를 만들 때도 불필요한 의존성을 끌고 가야 했습니다.

현재 구조에서는 Runtime target이 프로토콜과 무관한 TCP 런타임만 제공합니다. 애플리케이션은 필요한 target을 링크하고, 소스 파일에서는 필요한 public header만 include합니다.

CMake target은 모듈 단위로 링크
C++ include는 필요한 public header만 선택

2.2 모듈 target과 public header

이 문서의 본문은 iouring_runtime::Runtime이 제공하는 코어 런타임만 자세히 다룹니다.

3 Runtime core 구조

Runtime core는 다음 기능만 책임집니다.

  • io_uring 생성과 dispatch
  • listener accept flow
  • session lifecycle과 self-ownership
  • recv/send buffer 관리
  • send queue와 backpressure
  • timeout/watchdog hook
  • job queue, global queue, timer
  • ring 간 task 전달

HTTP request/response, TLS termination 정책, 게임 packet schema, DB, 도메인 로직은 core에 넣지 않습니다. 그 기능은 선택 모듈이나 예제 애플리케이션이 담당합니다.

4 io_uring 네트워크 엔진

IoRing은 io_uring 인스턴스를 RAII로 관리하고, 현재 스레드의 ring을 thread_local로 노출합니다. runtime core의 I/O 경로는 세 가지 기능을 중심으로 구성됩니다.

  • Multishot Accept: 하나의 accept SQE가 여러 연결 CQE를 발생시킵니다.
  • Multishot Recv + Provided Buffer: 커널이 buffer ring에서 수신 버퍼를 직접 선택합니다.
  • CQE batch dispatch: completion queue를 한 번에 수확해 event owner로 전달합니다.

4.1 Dispatch와 CQE batch 처리

Dispatch()는 CQE를 순회하면서 일반 I/O event와 ring 간 task event를 분기합니다. 일반 I/O event는 IoEvent*를 통해 owner를 찾고, task event는 user_data의 bit 0으로 구분합니다.

src/runtime/ring/IoRing.cpp — Dispatch() 구조
bool IoRing::Dispatch(std::chrono::milliseconds timeout) {
    std::vector<EventHandlerRef> keep_alive;
    unsigned head = 0;
    unsigned count = 0;

    io_uring_for_each_cqe(ring_, head, cqe) {
        const std::uint64_t data = cqe->user_data;
        if (data & 0x1ULL) {
            auto* task = reinterpret_cast<std::move_only_function<void()>*>(
                data & ~0x1ULL);
            (*task)();
            delete task;
        } else if (data != 0) {
            auto* ev = reinterpret_cast<IoEvent*>(data);
            auto owner = ev->Owner();
            if (owner) {
                keep_alive.push_back(owner);
                owner->Dispatch(ev, cqe->res, cqe->flags);
            }
        }
        ++count;
    }

    io_uring_cq_advance(ring_, count);
    return true;
}

keep_alive는 batch 순회가 끝날 때까지 handler의 shared_ptr를 잡아둡니다. 같은 batch 안에 같은 session의 CQE가 여러 개 있을 때, 앞쪽 CQE 처리 중 session이 release 조건을 만족하더라도 뒤쪽 CQE가 이미 파괴된 객체를 참조하지 않도록 막는 장치입니다.

4.2 Provided Buffer Ring

수신 경로에서는 IOSQE_BUFFER_SELECT와 provided buffer ring을 사용합니다. 유저스페이스가 매번 recv buffer 주소를 지정하지 않고, 커널이 미리 등록된 buffer 중 하나를 선택해 데이터를 기록합니다.

구현 포인트는 다음과 같습니다.

  • RingBuffer::Create()에서 metadata와 data 영역을 한 번에 mmap합니다.
  • 초기화 시 io_uring_buf_ring_add로 모든 buffer를 kernel ring에 등록합니다.
  • CQE 수신 후에는 buffer id를 확인하고, 사용한 buffer를 즉시 ring에 돌려줍니다.
  • ENOBUFS나 multishot 종료 flag를 만나면 recv를 재등록합니다.

TCP stream 특성상 패킷 단편화는 피할 수 없습니다. Runtime core는 수신된 byte span을 session에 전달하고, packet framing이 필요한 모듈은 별도 RecvBuffer로 조립합니다. 이 때문에 core는 HTTP나 game packet schema를 몰라도 됩니다.

5 Session 수명관리

io_uring 서버에서 가장 조심해야 할 부분은 객체 수명입니다. 애플리케이션이 session을 닫았다고 해서 커널에 제출한 SQE가 즉시 사라지는 것은 아닙니다. 이미 제출된 recv/send/ shutdown/cancel SQE는 나중에 CQE로 돌아올 수 있고, 그 CQE가 해제된 session 메모리를 참조하면 use-after-free가 발생합니다.

Runtime core는 이 문제를 세 겹으로 막습니다.

  1. self_ref_로 session이 자기 자신을 보유합니다.
  2. pending_io_로 아직 CQE를 받지 못한 in-flight I/O 수를 추적합니다.
  3. IoEvent는 owner를 weak_ptr로 잡고, dispatch batch 동안만 keep_alive로 승격합니다.

5.1 self_ref_와 pending_io_

include/iouring_runtime/core/Session.h — 개념 구조
class Session : public EventHandler,
                public std::enable_shared_from_this<Session> {
    std::shared_ptr<Session> self_ref_;
    int pending_io_ = 0;
    bool disconnecting_ = false;

    void AddPendingIo();
    void CompletePendingIo();
    void TryRelease();
};

수명 규칙은 단순합니다.

  • SQE 제출 시 pending_io_를 증가시킵니다.
  • CQE 처리 시 pending_io_를 감소시킵니다.
  • Disconnect() 후 pending_io_ == 0이 되면 self_ref_를 해제합니다.
  • self_ref_가 해제되는 시점에야 session 객체가 실제로 소멸할 수 있습니다.

이 구조에서는 session의 실제 파괴 시점이 애플리케이션 호출 순서가 아니라 커널 완료 신호에 의해 결정됩니다.

5.2 weak_ptr IoEvent와 batch keep_alive

IoEvent가 owner를 shared_ptr로 직접 잡으면 in-flight I/O가 session 수명을 필요 이상으로 늘릴 수 있습니다. 반대로 raw pointer만 저장하면 session이 먼저 사라지는 순간 위험합니다.

그래서 IoEvent는 owner를 weak_ptr로 저장합니다. CQE 처리 시점에만 승격을 시도하고, 승격에 실패하면 이미 논리적으로 종료된 객체이므로 해당 CQE는 무시합니다. 승격에 성공하면 Dispatch()의 local keep_alive vector가 batch 끝까지 owner를 붙잡습니다.

이 설계 덕분에 다음 두 요구를 동시에 만족합니다.

  • 종료된 session의 늦게 도착한 CQE를 안전하게 무시합니다.
  • batch 처리 중에는 살아 있는 owner가 중간에 파괴되지 않습니다.

6 Multi-thread 구조와 ring 간 task 전달

Runtime core는 ring-per-thread 모델을 사용합니다. 각 worker thread는 자기 IoRing, Listener, BufferPool을 가지고 이벤트 루프를 돕니다.

부하 분산과 cross-thread 작업에는 두 가지 장치를 사용합니다.

  • SO_REUSEPORT: 여러 listener가 같은 port에 bind되고, 커널이 connection을 worker에 분산합니다.
  • MSG_RING: 다른 ring의 completion queue에 task pointer를 직접 전달합니다.

6.1 MSG_RING task 전달

src/runtime/ring/IoRing.cpp — RunOnRing() 개념 구조
bool IoRing::RunOnRing(std::move_only_function<void()> task) noexcept {
    if (Current() == this) {
        task();
        return true;
    }

    auto* task_ptr = new std::move_only_function<void()>(std::move(task));
    std::uint64_t tagged =
        reinterpret_cast<std::uint64_t>(task_ptr) | 0x1ULL;

    io_uring_sqe* sqe = io_uring_get_sqe(source_ring);
    io_uring_prep_msg_ring(sqe, target_fd, 0, tagged, 0);
    sqe->flags |= IOSQE_CQE_SKIP_SUCCESS;
    io_uring_submit(source_ring);
    return true;
}

user_data의 bit 0은 “이 CQE는 task pointer”라는 tag입니다. 대상 ring의 Dispatch()는 bit 0을 보고 task를 복원해 실행합니다. 소스 ring에는 성공 CQE를 만들지 않기 위해 IOSQE_CQE_SKIP_SUCCESS를 사용합니다.

7 JobQueue, GlobalQueue, JobTimer

Runtime core의 job system은 “I/O thread 위에서 안전하게 순차 작업을 실행하는” 최소 구조입니다. 게임 서버 전용 기능이 아니라, 특정 객체의 작업을 한 번에 하나씩 처리하고 싶을 때 쓰는 범용 스케줄링 도구입니다.

7.1 JobQueue

JobQueue는 직렬 실행 큐입니다. 외부에서 작업을 Push()하면 큐에 쌓이고, 실제 실행은 Execute()를 호출한 worker thread에서 이루어집니다. 한 JobQueue는 동시에 하나의 thread에서만 실행되므로 queue owner 내부 상태에는 별도 lock을 덜 사용할 수 있습니다.

7.2 GlobalQueue

GlobalQueue는 실행 대기 중인 JobQueue를 worker들이 가져가는 중앙 큐입니다. 여러 worker가 TryPop()으로 대기 중인 queue를 가져가므로 자연스럽게 work stealing에 가까운 분배가 됩니다.

7.3 JobTimer

JobTimer는 지연 실행을 담당합니다. 지정한 시간이 지나면 callback을 바로 실행하지 않고, 대상 JobQueue에 작업으로 분배합니다. 이렇게 하면 timer callback도 queue owner의 직렬 실행 규칙 안에서 처리됩니다.

8 Core 직접 사용 예시

Core만 링크하면 HTTP나 game 모듈 없이 custom TCP 서버를 만들 수 있습니다.

find_package(iouring_runtime CONFIG REQUIRED)

add_executable(my_echo src/main.cpp)
target_link_libraries(my_echo PRIVATE
    iouring_runtime::Runtime
)
target_compile_features(my_echo PRIVATE cxx_std_23)
#include <iouring_runtime/core/IoRing.h>
#include <iouring_runtime/core/Listener.h>
#include <iouring_runtime/core/SendBuffer.h>
#include <iouring_runtime/core/Session.h>
#include <iouring_runtime/core/Types.h>

핵심 사용 흐름은 다음과 같습니다.

IoRing::Create
BufferPool 생성
Session subclass 정의
SessionFactory 작성
Listener::Start
while running:
    ring.Dispatch()
    ring.ProcessPostedTasks()
    GlobalQueue 작업 처리

대표 예제는 app/examples/runtime/core_echo와 app/examples/runtime/core_idle_echo입니다.

9 선택 모듈 요약

아래 모듈은 Runtime core 위에 올라간 사용 예시이자 배포 단위입니다. 이 문서에서는 세부 앱 구현보다 “core가 어떻게 확장되는지”만 보여줍니다.

9.1 RuntimeWeb

RuntimeWeb은 HTTP parser, router, middleware, request/response helper, file/streaming response를 제공합니다. 예제 앱은 hello_http, file_store_server, dropapp, speedtest_server, status_server, webhook_inbox입니다.

target_link_libraries(my_web_app PRIVATE
    iouring_runtime_web::RuntimeWeb
)

RuntimeWeb은 core의 Listener, Session, IoRing, timeout/backpressure 구조를 사용하지만, HTTP routing과 response 생성은 web 모듈의 책임으로 분리합니다.

9.2 RuntimeProxy

RuntimeProxy는 HTTP layer를 거치지 않는 TCP reverse proxy 모듈입니다. downstream session과 upstream connector를 bridge하고, 필요하면 downstream TLS termination, SNI route, ACME challenge responder, metrics snapshot을 제공합니다.

target_link_libraries(my_proxy PRIVATE
    iouring_runtime_proxy::RuntimeProxy
)

대표 실행 앱은 app/examples/proxy/tcp_reverse_proxy입니다.

9.3 RuntimeGame

RuntimeGame은 binary packet server를 위한 공통 구조만 제공합니다. PacketSession, PacketBuilder, PlayerRegistry, Room, RoomManager가 여기에 들어가고, 인증/전투/던전/DB는 예제 애플리케이션인 dungeon_full_server 쪽에 남깁니다.

target_link_libraries(my_game PRIVATE
    iouring_runtime_game::RuntimeGame
)

이 분리 덕분에 game module은 “패킷 서버 공통 계층”으로 유지되고, 실제 게임 도메인은 core나 module target을 오염시키지 않습니다.

10 검증과 예제

현재 저장소는 core와 선택 모듈을 함께 켜서 빌드·테스트할 수 있습니다.

cmake -S . -B build-all \
  -DCMAKE_BUILD_TYPE=Release \
  -DBUILD_WEB=ON \
  -DBUILD_PROXY=ON \
  -DBUILD_GAME=ON \
  -DBUILD_TESTS=ON \
  -DBUILD_EXAMPLES=ON
cmake --build build-all -j$(nproc)
ctest --test-dir build-all --output-on-failure

주요 테스트 범위:

  • core queue와 ring 동작
  • listener/session lifecycle
  • partial send와 inactivity timeout
  • cross-thread task flow
  • HTTP parser/router/response framing
  • TCP proxy smoke test
  • game packet/session/room helper
  • media와 observability helper
