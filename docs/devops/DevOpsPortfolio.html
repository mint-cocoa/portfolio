홈랩 DevOps 포트폴리오

io_uring C++ 런타임으로 만든 웹 앱 배포 파이프라인

저자

배진후

개요

이 문서는 iouring-runtime 위에서 실제 홈랩 서비스를 만들고, 컨테이너 이미지, GitHub Actions, GitOps, Argo CD, Kubernetes Ingress까지 연결한 과정을 정리한 운영 포트폴리오입니다. 목표는 단순한 예제 서버가 아니라, 개인 도메인에서 접근 가능한 실사용 웹 앱을 C++ 런타임 기반으로 배포하는 것이었습니다.

구현한 서비스는 두 가지입니다.

  • dropapp: 1시간 TTL을 가진 파일 드롭 앱
  • webhook-inbox: GitHub Actions, Argo CD, 홈랩 서비스 이벤트를 받는 운영 이벤트 inbox

문제 정의

기존 포트폴리오 문서는 게임 서버와 클라이언트 구현 중심이었습니다. 하지만 서버 런타임을 실제 운영 환경까지 연결했다는 증거는 별도로 정리되어 있지 않았습니다. 이번 작업의 핵심 질문은 다음과 같았습니다.

  1. 직접 만든 C++ io_uring 런타임이 HTTP 앱을 안정적으로 서빙할 수 있는가?
  2. 앱을 이미지화하고, GHCR, GitOps, Argo CD로 자동 배포할 수 있는가?
  3. 배포 결과를 내 도메인에서 실제 서비스로 노출할 수 있는가?
  4. 운영 중 이벤트와 상태를 모아볼 수 있는 내부 가시성 앱을 만들 수 있는가?

전체 구조

배포 흐름은 애플리케이션 코드와 운영 매니페스트를 분리했습니다.

  • 앱 코드는 mint-cocoa/iouring-runtime 또는 mint-cocoa/portfolio에 둡니다.
  • 컨테이너 이미지는 ghcr.io/mint-cocoa/<app>:<GITHUB_SHA>로 푸시합니다.
  • GitHub Actions가 GitOps repo의 Helm values를 해당 SHA로 갱신합니다.
  • Argo CD가 변경된 chart를 감지해 클러스터에 반영합니다.

C++ 런타임 기반 웹 앱

iouring_runtime_web는 WebServer, 라우터, request context, response builder, streaming upload API를 제공합니다. 웹 앱은 다음과 같은 형태로 작성했습니다.

iouring_runtime::web::WebServer server(config);

server.Get("/healthz", [](RequestContext& ctx) {
    ctx.response.ContentType("text/plain; charset=utf-8")
        .Body("ok")
        .Send();
});

server.Get("/", [&](RequestContext& ctx) {
    ServeStaticFile(ctx, static_root, "index.html");
});

중요한 점은 단순히 TCP 서버를 만든 것이 아니라, HTTP 라우팅, 정적 파일 서빙, raw body upload, JSON API, 인증 헤더 처리까지 실제 웹 앱에 필요한 표면을 구현했다는 점입니다.

dropapp

dropapp은 TTL 파일 드롭 앱입니다.

항목 내용
업로드 POST /api/files, raw body streaming upload
파일명 X-Dropapp-Filename-Encoded 헤더 사용
저장 /data/objects/<id> + metadata
만료 기본 1시간 TTL
보안 optional bearer token for upload/delete
배포 dropapp.mintcocoa.cc

처음에는 파일명을 일반 헤더로 보내며 한글 파일명에서 브라우저 오류가 발생했습니다. 해결은 파일명을 encodeURIComponent로 인코딩해 전송하고, 서버에서 안전하게 디코딩하는 방식으로 처리했습니다.

webhook-inbox

webhook-inbox는 운영 이벤트 수집 앱입니다. 초기 버전은 범용 webhook capture에 가까웠지만, 이후 GitHub Actions, Argo CD, 홈랩 서비스 이벤트를 모으는 운영 타임라인 앱으로 확장했습니다.

주요 경로는 다음과 같습니다.

경로 역할
POST /hooks/github GitHub webhook 수신
POST /hooks/argocd Argo CD notification 수신
POST /hook/{service} 기타 홈랩 서비스 이벤트 수신
GET /api/events 최근 이벤트 조회
GET /api/events/{id} 원본 body/header 검사

저장은 append-only JSONL로 처리합니다.

{"service":"github","event_type":"workflow_run","status":"success","source":"mint-cocoa/iouring-runtime"}
{"service":"argocd","event_type":"sync","status":"Succeeded","source":"dropapp"}

앱 시작 시 JSONL을 다시 읽어 in-memory index를 구성합니다. 삭제는 기존 라인을 수정하지 않고 tombstone 라인을 append해서 재시작 후에도 삭제 상태가 유지되도록 했습니다.

컨테이너 이미지

각 앱은 multi-stage Dockerfile을 사용합니다.

FROM ubuntu:24.04 AS build
RUN apt-get update && apt-get install -y build-essential cmake ninja-build liburing-dev
COPY . .
RUN cmake -S . -B build -G Ninja -DBUILD_WEB=ON -DBUILD_EXAMPLES=ON
RUN cmake --build build --target webhook_inbox

FROM ubuntu:24.04 AS runtime
RUN apt-get update && apt-get install -y ca-certificates liburing2
COPY --from=build /src/build/bin/webhook_inbox /usr/local/bin/webhook_inbox

runtime image에는 빌드 도구를 넣지 않고, 실행 바이너리와 정적 파일만 복사합니다. io_uring 사용을 위해 런타임에는 liburing2만 필요합니다.

GitHub Actions

main push 시 workflow가 다음 순서로 실행됩니다.

  1. GHCR 로그인
  2. Docker image build/push
  3. home-k8s-gitops checkout
  4. Helm values의 image.tag를 ${GITHUB_SHA}로 갱신
  5. GitOps repo에 commit/push

이 구조 덕분에 앱 repo의 commit SHA가 그대로 배포 이미지 tag가 됩니다. 문제가 생겼을 때 Kubernetes에서 실행 중인 이미지 tag만 봐도 어떤 commit인지 추적할 수 있습니다.

GitOps와 Argo CD

GitOps repo는 앱별 Helm chart와 cluster application manifest를 분리했습니다.

apps/
  dropapp/
  webhook-inbox/
clusters/home/
  dropapp-application.yaml
  webhook-inbox-application.yaml

Application은 automated sync를 사용합니다.

syncPolicy:
  automated:
    prune: true
    selfHeal: true
  syncOptions:
    - CreateNamespace=true

이 설정으로 GitOps repo가 바뀌면 Argo CD가 자동으로 Deployment, Service, Ingress, PVC를 맞춥니다.

도메인과 Ingress

클러스터 내부에서는 nginx ingress와 MetalLB를 사용했습니다.

도메인별 라우팅은 Ingress host로 분리합니다.

  • dropapp.mintcocoa.cc
  • webhook.mintcocoa.cc
  • portfolio.mintcocoa.cc

외부에서 실제 접속하려면 edge proxy의 host route도 동일하게 클러스터 ingress LoadBalancer IP로 연결해야 합니다.

운영 검증

배포 후 다음을 검증했습니다.

kubectl get application -n argocd
kubectl get all,ingress,pvc -n webhook-inbox
curl --resolve webhook.mintcocoa.cc:80:172.30.1.240 \
  http://webhook.mintcocoa.cc/healthz

검증 포인트는 단순히 Pod가 뜨는지가 아니라, 실제 Ingress 경로로 HTTP API가 응답하고, 이벤트가 PVC에 JSONL로 기록되는지까지 확인하는 것이었습니다.

결과

이번 작업으로 다음을 달성했습니다.

  • 직접 만든 C++ io_uring 웹 런타임으로 실사용 앱 2개를 구현했습니다.
  • 앱을 GHCR 이미지로 빌드하고 GitOps repo를 자동 갱신했습니다.
  • Argo CD automated sync로 홈 Kubernetes 클러스터에 배포했습니다.
  • 도메인 기반 Ingress 라우팅과 /healthz 검증 경로를 갖췄습니다.
  • 운영 이벤트를 수집하는 내부 가시성 앱을 추가했습니다.
  • 이 포트폴리오 문서 자체도 같은 런타임 서버 위에 서빙할 수 있게 구성했습니다.

회고

가장 큰 성과는 런타임, 앱, 컨테이너, GitOps, 도메인 라우팅이 하나의 end-to-end 시스템으로 연결됐다는 점입니다. C++ 서버 구현만으로 끝내지 않고, 실제 운영에 필요한 배포 자동화와 관측 가능성까지 연결하면서 런타임의 실사용성을 검증했습니다.

다음 단계는 webhook-inbox에 내부 service health poller를 추가해 webhook 기반 이벤트뿐 아니라 서비스의 현재 상태도 자동으로 수집하는 것입니다.
