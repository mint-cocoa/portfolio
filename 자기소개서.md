# [직무] 개발자로서의 성장과 기여
안녕하세요, 저는 게임 개발에 깊은 관심을 가지고 있는 소프트웨어 전공 신입 지원자 배진후 입니다. 학부 시절 웹 서비스를 만들며 백엔드 개발에 흥미를 가지게 되었고, 여러 사용자가 동시에 접속하는 환경에서 서버의 안정적인 설계하며(servercore → v3 → v4 → libiouring-server) epoll 기반에서 io_uring 기반 프레임워크로 발전시키는 과정을 거쳤습니다. 매번 이전 버전의 병목을 측정하고, 그 원인을 분석해 아키텍처 자체를 바꾸는 경험을 반복하면서 개발자로서의 방향을 분명히 잡게 되었습니다.

많은 사용자가 동시에 접속하는 환경에서 성능과 안정성을 함께 요구하는 문제를 가장 치열하게 다루는 분야가 바로 게임 서버와 실시간 클라이언트 영역이라고 생각하기 때문입니다. 저는 단순히 기능을 구현하는 데서 그치지 않고, 왜 느린지, 왜 불안정한지, 어떤 구조가 더 나은지 끝까지 검증하는 개발자가 되고자 합니다. 입사 후에는 실시간 서버 구조, 네트워크 처리, 게임 플레이를 뒷받침하는 클라이언트 시스템을 깊이 있게 이해하며 서비스 품질 향상에 기여하고 싶습니다.

가장 중요하게 소개드리고 싶은 경험은 io_uring 기반 C++ 게임 서버 프레임워크를 직접 설계하고 구현한 프로젝트입니다. epoll과 io_uring의 차이를 실제 코드와 벤치마크로 검증하는 데서 시작해, 여러 차례 재설계를 거치며 성능과 안정성을 개선했습니다. 세부 구조와 구현 내용은 포트폴리오 문서에 정리해두었으며, 자기소개서에서는 그 과정에서 어떤 문제를 어떻게 해결했는지를 중심으로 말씀드리고자 합니다.

개발 과정에서 가장 어려웠던 문제는 대량 접속과 해제를 반복하는 환경에서 발생한 Use-After-Free였습니다. 세션이 먼저 파괴된 뒤 커널의 I/O 완료가 뒤늦게 도착하면서 이미 해제된 메모리에 접근하는 상황이었고, 이를 해결하기 위해 pending I/O 카운트와 self reference 패턴을 도입해 객체 수명 관리를 구조적으로 수정했습니다. 이후 Sanitizer, 테스트 코드, 로드테스트를 통해 안정성을 검증했고, Echo 서버 기준 초당 327K 처리와 Separated 아키텍처 기준 3.1M msg/s 처리량을 확인했습니다. 이 경험을 통해 성능 개선은 단순 튜닝이 아니라 문제 재현, 원인 분석, 구조 수정, 검증까지 이어져야 한다는 점을 분명히 배웠습니다.

클라이언트 측에서도 엔진 없이 DirectX 11 API만으로 아이소메트릭 멀티플레이 던전 RPG를 직접 구현하며 서버와 맞물리는 전체 흐름을 경험했습니다. 씬 전환, 네트워크 프로토콜, 렌더링 파이프라인 등 구현 내용은 포트폴리오에 별도로 정리해두었고, 자기소개서에서는 이 경험을 통해 서버뿐 아니라 클라이언트에서도 병목을 계층적으로 분석하는 시각을 갖게 되었다는 점을 강조하고 싶습니다. 실제로 던전 렌더링 구조를 인스턴싱 방식으로 바꾸며 Draw Call을 4,640개에서 8회 수준으로 줄였고, 수치 기반으로 개선 효과를 검증했습니다.

저의 강점은 문제를 재현하고, 측정하고, 구조 단위로 해결하는 집요함이라고 생각합니다. 새로운 기술을 접할 때도 문서만 읽고 끝내지 않고 직접 코드를 읽고 실험으로 이해하려고 하며, Nakama(Go 기반 오픈소스 게임서버)와 Unreal Engine 소스를 직접 fork하여 내부 구조를 분석해본 경험도 있습니다. 팀 프로젝트에서는 FastAPI, Kubernetes, GitHub Actions, ArgoCD를 활용한 MSA 기반 졸업작품을 진행했는데, API Gateway, User Service, Reservation Service 등 서비스별 저장소를 분리하고 91회 이상의 커밋을 거치며 실제 협업 과정을 경험할 수 있었습니다. API 문서 자동화와 배포 자동화를 도입하면서 팀원 간 작업 충돌을 줄였고, 기술 선택이 결국 협업 효율과 유지보수성으로 이어진다는 점을 체감했습니다.

입사 후에는 먼저 조직의 개발 방식과 서비스 특성을 빠르게 학습한 뒤, 맡은 기능을 안정적으로 구현하는 개발자가 되겠습니다. 서버 엔진부터 클라이언트 렌더링, 네트워크 프로토콜, 비동기 데이터베이스 연동까지 전체 파이프라인을 직접 구현해 본 경험이 있기에, 문제가 어느 계층에서 발생하는지 빠르게 파악하고 근거 있는 개선안을 제시할 수 있을 것이라 생각합니다. 더 나아가 Sanitizer와 프로파일러, 체계적인 로드테스트를 활용해 실제 지표로 증명하는 방식으로 서비스 품질 향상에 기여하고 싶습니다. [회사명]에서 사용자 경험을 뒷받침하는 견고한 시스템을 만드는 데 함께하고 싶습니다.

## 짧은 버전

안녕하세요. 성능과 안정성 문제를 구조적으로 해결하는 과정에서 보람을 느끼는 신입 개발자 배진후입니다. 학부 시절 웹 서비스를 만들며 백엔드 개발에 흥미를 가졌고, 동시 접속 환경의 병목과 최적화에 관심을 갖게 되었습니다. 또한 게임 서버는 단순한 API 기반의 요청-응답 구조를 넘어서 사용자 간 동기화, 상태 관리, 멀티스레딩 처리 등 복잡한 기술적 과제가 많다는 점이 인상깊었습니다. 이러한 복합적인 문제들을 해결하는 과정에서 성취감을 느꼈고, 기능을 구현하는 데 그치지 않고 효율성과 안정성을 동시에 고려해야 한다는 점에서 서버 개발이 제게 가장 잘 맞는 분야라고 생각했습니다.

포트폴리오엔 두가지,   C++ 게임 서버 프레임워크를 설계하고 구현한 프로젝트입니다. epoll과 io_uring의 차이를 실제 코드와 벤치마크로 검증, 여러 차례 재설계를 거치며 성능과 안정성을 개선하기 위해 노력했습니다. 

두번째 프로젝트로는 DirectX 11 기반 멀티플레이 RPG 클라이언트를  구현하며 서버와 클라이언트를 함께  시각을 갖추었습니다. 던전 렌더링 구조를 인스턴싱 방식으로 재설계해 Draw Call을 4,640개에서 8회 수준으로 줄이며 성능 개선 효과를 수치로 확인했습니다.


졸업작품으로 진행했던 팀 프로젝트에서는 협업 능력 및 개발 도구 자체의 숙련도를 기르기 위해  서비스 성격에 맞춰 서비스를 모듈화해 구현해 나가는  졸업작품을 진행했는데, API Gateway, User Service, Reservation Service 등 서비스별 저장소를 분리하고 개발 내부 프로세스로는 github action 및 argocd 를 이용해 git 부터 배포까지의 자동화를 구현하고자 했습니다. 이를 통해  기술적인 부분 뿐만 아니라 협업과 커뮤니케이션의 중요성도 함께 인식 할수 있었습니다.

신입 개발자로서 부족한 실무 경험을 빠르게 채워나가기 위해 노력과 학습을 통해 역량을 키워나갈 것이며, 맡은 업무에 책임감을 가지고 성실하게 임하겠습니다. 또한 팀원들과의 협업과 커뮤니케이션을 중요시하며, 함께 성장하는 개발자가 되겠습니다. 감사합니다.

저는 학부 졸업 작품을 웹 서버 개발로 시작하면서 백엔드와 서버 개발에 흥미를 갖게 되었습니다. 프로젝트를 진행하며 여러 사용자가 동시에 접속하는 상황을 고려한 부하 테스트를 수행했고, 이를 통해 서버의 성능 최적화와 안정성 확보에 대한 관심이 자연스럽게 생겼습니다. 이 과정에서 단순한 웹 서비스보다 더 높은 성능과 실시간성이 요구되는 게임 서버 개발에 큰 매력을 느끼게 되었습니다.

특히 게임 서버는 단순한 API 기반의 요청-응답 구조를 넘어서 사용자 간 실시간 상호작용, 동기화, 상태 관리, 지연 시간 최소화, 멀티스레딩 처리 등 복잡한 기술적 과제가 많다는 점이 인상깊었습니다. 이러한 복합적인 문제들을 해결하는 과정에서 성취감을 느꼈고, 기능을 구현하는 데 그치지 않고 효율성과 안정성을 동시에 고려해야 한다는 점에서 서버 개발이 제게 가장 잘 맞는 분야라는 확신을 갖게 되었습니다.

프로젝트를 진행할 때 팀원의 개발 편의성과 코드 관리의 효율성을 중요하게 생각합니다. 졸업작품으로 진행한 프로젝트에서는 예약, 티켓 발급, 시설 관리, 실시간 위치 추적, 알림 등 테마파크의 다양한 서비스를 독립적인 msa 분리하여 개발했습니다.
ArgoCD와 GitHub Actions를 이용해서 코드 변경 사항이 Git에 반영되면 자동으로 빌드 후 개발용 쿠버네티스 클러스터에 배포되도록 설정하였습니다. 이를 통해 각 팀원의 환경 차이로 인한 호환성 문제를 최소화했고, 백엔드 서버는 FastAPI로 구현하여 API 문서를 Swagger를 통해 자동화해 팀 간 연동을 원활하게 만들었습니다. 

또한  소켓 프로그래밍에 대한 이해를 높이기 위해 Linux 환경에서 epoll과 io_uring을 활용한 에코 서버를 직접 구현해보았습니다. 
io_uring API의 작동 원리와 성능적 이점에 대해 관심을 갖고 분석했으며 이를 epoll 기반 서버와 비교 분석하는 프로젝트를 진행했습니다.  프로젝트 진행을 위해서 소켓 프로그래밍 기본 개념 및 동시 접속 처리, 메시지 전달, 연결 관리 등의 네트워크 프로그래밍 기본 구조를 이해하기 위해 클라이언트와 서버간 통신 구조체 설계, 소켓 생성부터 소켓 래핑 클래스까지 직접 구현해서 사용했습니다.
그리고 처리 능력을 극대화시키기 위해 멀티쓰레딩 방식을 적용했습니다. 각 cpu 코어별 로 io 처리를 위한 인스턴스를 독립적으로 배치하고, 이를 관리하는 매니저 클래스를 통해 동시성 문제를 관리했고, 오픈소스로 구현된 벤치마킹을 코드를 수정해 테스트를 진행했습니다. 
이를 통해 TCP 기반 통신 구조, 클라이언트 연결 방식, 세션 관리 등 서버 시스템의 핵심 개념을 체득할 수 있었고, 두 가지 I/O 모델의 성능을 측정하고 비교 분석하며 각각의 장단점과 실제 활용 시 고려할 요소들에 대해 구체적으로 이해할 수 있었습니다. 분석 보고서는 개인 사이트에 호스팅해서 보기 좋게 구성했습니다.

이러한 경험들은 저로 하여금 기술적인 부분 뿐만 아니라 협업과 커뮤니케이션의 중요성도 함께 인식하게 해주었고, 그 과정에서 서버 개발자로서의 역량과 책임감을 키워나갈 수 있었습니다. 앞으로도 복잡한 문제를 기술적으로 해결하는 동시에  팀과의 긴밀한 협력을 통해 더 나은 게임을 만들어 나가는 서버 개발자가 되고자 합니다.
